# pbflags v0.16.0

Config-driven flags: flag behavior is now defined in YAML config files with CEL
condition chains, replacing the per-entity override system. The admin UI is
read-only â€” the kill switch is the only runtime control.

See [Upgrading: v0.16.0](../upgrading.md#conditions-and-config-driven-flags-v0160)
for detailed migration steps.

## Breaking Changes

- **Overrides removed.** The `flag_overrides` table is dropped. `SetFlagOverride`
  and `RemoveFlagOverride` RPCs return `Unimplemented`. Export existing database
  flag values and per-entity overrides with `pbflags-sync export` before upgrading.
- **State narrowed.** `flags.state` (ENABLED/DEFAULT/KILLED) replaced by
  `killed_at TIMESTAMP NULL`. Only the kill switch remains as a runtime control.
- **`flags.value` and `flags.layer` dropped.** Values are determined by the
  `conditions` JSONB column, set by the sync pipeline from YAML config files.
- **Admin UI is read-only.** Value editing and override management removed.
  Condition chains display as a table; kill/unkill buttons remain.
- **`layerutil` package removed.** Layer codegen paths and `(pbflags.layers)`
  extension are gone. If your proto still defines a layers enum, the codegen
  will ignore it.

## Server

- Condition evaluation engine with CEL (Common Expression Language) support
- Dimension-classified cache keys with LRU condition caching
- Observability: condition evaluation metrics, flag_id in CEL error logs
- Shared `flagfmt` package: `StoredCondition` type, `DisplayString`/`AsAny` formatters
- Killed flags return typed zero values instead of discarding the condition chain
- Defensive `LEFT JOIN` and `jsonb_typeof` guards in admin queries

## Configuration Management

- YAML config files define flag conditions and static values per feature
- `pbflags-sync --features=./features` compiles YAML into database conditions
- `pbflags-sync --sha=<hash>` records the git commit (displayed in admin UI)
- `pbflags-sync validate` checks syntax, CEL compilation, and types without touching the DB
- `pbflags-sync show <flag>` renders a flag's condition chain for inspection
- `pbflags-sync export` generates YAML from existing database flag values and
  legacy per-entity overrides (migration bridge)
- `pbflags-admin --standalone --features=./features` syncs conditions on startup
- `.pbflags.yaml` project config: set `features_path` once, CLI infers `--features` automatically

## Admin UI

- Config-aware dashboard: shows condition count per flag, killed badge, no more "all defaults"
- Flag detail: condition chain table (CEL expression -> value), sync SHA badge
- Read-only value display; list types render as chips
- Conditions error banner surfaces JSON unmarshal failures
- Removed: value editing, override management, bulk import, layer filter

## Bug Fixes

- Fixed `safeSlice` template helper: complete clamping for negative/inverted indices
- Fixed `GetFlag` JOIN: LEFT JOIN preserves flags when features row is missing
- Fixed `jsonb_array_length`: guards against non-array conditions JSONB
- Fixed static YAML values: `value: X` entries now sync to database (stored as single-entry condition chains)
- Fixed CEL eval logging: includes flag_id, condition index, and expression source
- Deduplicated three condition JSON structs and three FlagValue formatters into `flagfmt`
- Fixed Makefile `dev/descriptors.pb` prerequisite: uses `find` for recursive proto discovery
- Demo config uses symbolic `PlanLevel.ENTERPRISE` instead of numeric ordinals
