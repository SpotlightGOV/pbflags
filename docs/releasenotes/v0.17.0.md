# pbflags v0.17.0

Launches v1: gradual rollouts with per-condition value overrides and evaluation scopes.

## Breaking changes

- **Proto**: `DimensionOptions.hashable` (bool) replaced by `DimensionDistribution` enum (wire-compatible: old `hashable: true` decodes as `UNIFORM`). `bounded` (bool, field 3) removed (reserved). New `DimensionPresence` enum on field 4.
- **Proto**: `CompiledFeature.launches` (field 6) reserved. Launches move to `CompiledBundle.launches` (field 3). `CompiledLaunch` drops `flag_id`, `population_cel`, `value_json`; adds `scope_feature_id`, `affected_features`, `description`.
- **DB**: Migration 008 reshapes the `launches` table — drops `flag_id`, `population_cel`, `value` columns; renames `feature_id` → `scope_feature_id` (nullable); adds `affected_features`, `description`, `killed_at`. Status value `BAKED` → `SOAKING`. Precondition: no ACTIVE/BAKED launches.
- **YAML config**: Launch definitions change from `{flag, dimension, population, value, ramp_percentage}` to `{dimension, ramp_percentage, description}`. Launch-to-flag binding moves inline to conditions via `launch: {id, value}`.

See the [upgrade guide](../upgrading.md#launches-and-evaluation-scopes-v0170) for migration steps.

## Launches

- Per-condition value overrides: a launch changes what individual conditions return, not the structure of the condition chain. Each condition may have at most one `launch: {id, value}` override.
- Static value overrides: launches can also override non-conditional flag values.
- Feature-scoped launches defined in the feature's YAML config; cross-feature launches in a `features/launches/` subdirectory (ID = filename).
- Launch kill switch (`killed_at`) orthogonal to lifecycle status — killing a SOAKING launch and unkilling it returns to SOAKING.
- Stale launch cleanup: launches removed from config are marked ABANDONED by both `SyncConditions` and `LoadBundle`.

## Evaluation scopes

- New `(pbflags.scope)` file-level proto option defines named execution contexts with dimension sets.
- `FeatureOptions.scopes` declares which scopes a feature is available in.
- Codegen produces per-scope `*Features` types (e.g., `AnonFeatures`, `UserFeatures`) with constructors requiring scope dimensions as typed parameters.
- Feature accessor methods only exist on scopes the feature lists — wrong-scope access is a compile error.
- Duck-typed `Has<Feature>` interfaces per feature for handler decoupling.
- Context storage helpers: `ContextWith<Scope>Features` / `<Scope>FeaturesFrom`.

## Evaluator

- Launch evaluation is inline within the condition chain — no separate `EvaluateLaunches` step.
- `EvaluationSource_EVALUATION_SOURCE_LAUNCH` returned when a launch override is applied (visible in metrics, tracing, source labels).
- Empty dimension values are hashed (not skipped) — 100% ramp is behavior-preserving for `pb launch land`.
- DB fetcher extracts launch IDs from compiled conditions and batch-fetches active/soaking unkilled launches.

## Validation

- Unified `CollectLaunches` used by `SyncConditions`, `Compile`, and `Validate` — all three paths enforce the same rules: reference existence, scope enforcement, duplicate ID detection, UNIFORM dimension check, affected_features collection.
- `pbflags-sync validate` now catches launch reference errors before deploy.

## Admin UI

- Conditions table gains "Launch Override" column showing launch ID and override value.
- Launches section on flag detail shows scope, dimension, ramp, status, killed badge.
- Kill/unkill buttons per launch via `POST /api/launches/kill/{launchID}` and `/unkill/{launchID}`.

## Lint

- `scope_removed`: removing a scope deletes the generated `*Features` type.
- `scope_dimension_changed`: changing a scope's dimension set changes the constructor signature.
- `feature_scope_removed`: removing a feature from a scope deletes the generated accessor.
