# Launches

**Status:** Draft
**Date:** 2026-04-14
**Author:** bmt
**Builds on:** [Evaluation Context and Conditional Flags](2026-04-12-evaluation-context-and-conditions.md)

## Problem

pbflags has conditions (CEL-based value selection) and static values, but no
mechanism for gradually rolling out a value change. Today, changing a flag value
means editing the YAML config and syncing — the change takes effect for 100%
of traffic immediately. There's no way to say "roll this change out to 10% of
users, observe, then increase."

The existing launch implementation (migration 007, `evaluator/launch.go`)
stores a flat value per launch and evaluates it independently of the condition
chain. This design replaces that model with one that integrates launches into
the condition chain itself, supports multi-feature coordination, and provides
a mechanical path from "ramping" to "landed."

## Terminology

**Launch** — the unit of gradual rollout. A launch has a dimension (for
hashing), a ramp percentage (for traffic splitting), and a lifecycle
(CREATED → ACTIVE → SOAKING → COMPLETED / ABANDONED, with reversible
KILLED state). A launch may affect
one flag (feature-scoped) or many flags across features (cross-feature).

The word "launch" was chosen over "rollout" because launches are more than
a mechanical ramp — they're the organizational unit for shipping changes.
Future work adds approval gates, staged ramp plans, and review workflows
to the launch concept. See "Future work" at the end of this document.

## Goals

1. **Per-condition value overrides** — a launch changes what specific
   conditions return, not the structure of the condition chain.
2. **Multi-feature launches** — a single launch can affect flags across
   multiple features, ramping them together.
3. **Inline visibility** — reading a feature's config shows exactly how
   a launch affects it. No indirection to a separate launch definition
   to understand behavior.
4. **Mechanical landing** — `pb launch land <launch>` transforms the YAML
   config to promote launch values to defaults with guaranteed correctness.
   In Phase 1, the transform is a pure YAML operation — no CEL reasoning
   required. Phase 2 extends the same command with CEL simplification
   for `launch.in_ramp()` expressions (see "Phase 2" below).
5. **Bounded caching** — launches do not add unbounded cardinality to
   the condition cache. The existing bounded-output principle applies.

## Non-Goals

- **Condition chain replacement** (Option A from design discussion).
  Replacing the full chain is redundant and creates silent divergence risk
  when the base chain is edited during a launch. Per-condition overrides
  avoid this class of bug.
- **Launch state in CEL** (Option C). Deferred to Phase 2. Requires a
  CEL simplifier for mechanical landing. See "Phase 2" below.
- **Approval gates and staged ramp plans.** Future work on the launch
  lifecycle. The primitive ships first.
- **Experiments** (A/B testing with variant assignment and logging).
  Orthogonal; depends on launch infrastructure but has its own design
  surface.

## Design

### YAML config format

#### Feature-scoped launch (single feature)

When a launch affects only one feature, define it in the feature's own
config file:

```yaml
# features/notifications/notifications.yaml
feature: notifications

launches:
  daily-digest-rollout:
    dimension: user_id
    ramp_percentage: 25

flags:
  digest_frequency:
    conditions:
      - when: 'ctx.plan == PlanLevel.ENTERPRISE'
        value: "daily"
      - when: 'ctx.plan == PlanLevel.PRO'
        value: "weekly"
        launch:
          id: daily-digest-rollout
          value: "daily"
      - otherwise: "weekly"
```

The `launches:` section defines the launch — its hash dimension and initial
ramp percentage. The `launch:` key on individual conditions defines what
value that condition returns when the entity is in the launch's ramp.
Each condition may have at most one launch override (see "Single override
per condition" below).

In this example: Pro users in the 25% ramp get `"daily"` instead of
`"weekly"`. Enterprise users are unaffected (their condition has no launch
override). Users outside the ramp evaluate the condition chain normally.

**Scoping rule:** A launch defined in a feature config file can only be
referenced by conditions in that same file. The sync tool errors if a
feature-scoped launch is referenced from another feature:

```
error: flag "billing/3" references launch "daily-digest-rollout" which is
  scoped to feature "notifications". Move the launch to launches/ to use
  it across features.
```

#### Cross-feature launch (multiple features)

When a launch affects multiple features, define it in a standalone file
under a top-level `launches/` directory:

```
features/
  notifications/
    notifications.proto
    notifications.yaml
  billing/
    billing.proto
    billing.yaml
launches/
  pro-v2.yaml
```

```yaml
# launches/pro-v2.yaml
dimension: user_id
ramp_percentage: 10
```

The launch ID is the filename sans extension: `pro-v2.yaml` defines launch
`pro-v2`. No `launch_id:` field inside the file.

Referenced from multiple feature configs:

```yaml
# features/notifications/notifications.yaml
feature: notifications

flags:
  digest_frequency:
    conditions:
      - when: 'ctx.plan == PlanLevel.PRO'
        value: "weekly"
        launch:
          id: pro-v2
          value: "daily"
      - otherwise: "weekly"
```

```yaml
# features/billing/billing.yaml
feature: billing

flags:
  monthly_price:
    conditions:
      - when: 'ctx.plan == PlanLevel.PRO'
        value: 29.99
        launch:
          id: pro-v2
          value: 19.99
      - otherwise: 9.99
```

Both flags ramp together because they share the same launch. `pb launch ramp
pro-v2 50` moves both to 50%.

#### Static value overrides

Launches can also override static (non-conditional) flag values:

```yaml
flags:
  email_enabled:
    value: false
    launch:
      id: pro-v2
      value: true
```

This is sugar for "the default value changes under the launch." The
mechanical landing transform replaces `value: false` with `value: true`
and removes the `launch:` key.

#### Single override per condition

A condition (or static value) may have at most **one** launch override.
The sync tool errors if multiple are present:

```
error: flag "notifications/5" condition 2 has multiple launch overrides.
  Use Phase 2 launch.in_ramp() in CEL for overlapping launch semantics.
```

This constraint keeps Phase 1 genuinely mechanical:

1. Replace `value` with the launch override value.
2. Remove the `launch:` key.
3. Remove the launch definition.

No priority ordering, no overlap reasoning, no hidden behavior change.

For cases that need overlapping launches, intersections, or explicit
fallback ordering, Phase 2 `launch.in_ramp()` is the right tool because
the interaction is visible in the condition chain:

```yaml
conditions:
  - when: 'launch.in_ramp("pro-v2") && launch.in_ramp("digest-exp") && ctx.plan == PlanLevel.PRO'
    value: "realtime"
  - when: 'launch.in_ramp("pro-v2") && ctx.plan == PlanLevel.PRO'
    value: "daily"
  - when: 'launch.in_ramp("digest-exp") && ctx.plan == PlanLevel.PRO'
    value: "realtime"
  - when: 'ctx.plan == PlanLevel.PRO'
    value: "weekly"
```

More verbose, but appropriately so: overlapping rollout semantics are
subtle and should be explicit in reviewed config, not hidden in map
iteration order.

### Launch definition fields

| Field | Required | Description |
|-------|----------|-------------|
| `dimension` | Yes | Dimension for ramp assignment. Must have `distribution: UNIFORM`. Must be in the dimensions of every scope across all affected features. See "Dimension presence". |
| `ramp_percentage` | No | Initial ramp (0–100). Only applied on first sync (insert). Subsequent syncs preserve the operator-set ramp in the database. Default: 0. |
| `description` | No | Human-readable description for admin UI and CLI. |

Note the absence of `population` and `flag`. These fields from the current
implementation are unnecessary in this design:

- **`population`** is unnecessary because the condition's `when` clause
  already gates who sees the value. The launch doesn't need its own
  population filter — the condition chain handles targeting.
- **`flag`** is unnecessary because the launch-to-flag binding is expressed
  inline via the `launch:` key on individual conditions, not on the launch
  definition.

### Evaluation semantics

The evaluation precedence chain becomes:

```
1. Kill check (unchanged)
2. Condition chain evaluation (with launch overrides)
3. Static value / compiled default (unchanged)
```

Launches are no longer a separate evaluation step before conditions. Instead,
they modify what individual conditions return:

```
for each condition in chain:
    if condition.when matches context (or is "otherwise"):
        if condition has a launch override:
            if launch is ACTIVE or SOAKING:
                if entity is in launch ramp (hash check):
                    return launch override value  → source: LAUNCH
        return condition.value  → source: CONDITION
return compiled default  → source: DEFAULT
```

This preserves the "first condition match wins" invariant. The launch
override only applies within a matched condition — it doesn't change which
condition matches. Because each condition has at most one launch override,
there is no priority ambiguity.

### Launch hashing

The existing `EvaluateLaunches` implementation guards against empty
dimension values: `dimValue != "" && HashBucket(...)`. This guard was
defensive for optional dimensions where empty meant "not present." With
the scope model guaranteeing dimension presence, this guard is removed.

Launch hashing includes all dimension values, including empty strings:

```go
// Before (current code):
if dimValue != "" && HashBucket(launch.LaunchID, dimValue) < launch.RampPct {

// After:
if HashBucket(launch.LaunchID, dimValue) < launch.RampPct {
```

`HashBucket("launch-id", "")` is deterministic — an entity with an empty
dimension value always gets the same bucket. At 100% ramp, `bucket < 100`
holds for all buckets (0–99). This is what makes `pb launch land`
behavior-preserving without requiring generated constructors to validate
non-empty values.

### Kill semantics

Kill behavior follows the hierarchy from the evaluation-context design doc:

| Kill target | Effect | Reversible |
|-------------|--------|------------|
| Flag | All evaluations return compiled default. All launches on the flag are bypassed. | Yes (`pb unkill <flag>`) |
| Launch | All entities evaluate the base condition value. Other launches on the flag are unaffected. | Yes (`pb launch unkill <launch>`) |

Killing a launch is the common operational action. Killing a flag is the
nuclear option.

**Kill is orthogonal to lifecycle, not a status.** Launch kill state is
modeled as a separate `killed_at` column (nullable `TIMESTAMPTZ`), the
same pattern as flag kills. The `status` column continues to track
lifecycle (CREATED → ACTIVE → SOAKING → COMPLETED / ABANDONED)
independently. This avoids the "which state to restore" problem — killing
a SOAKING launch and unkilling it returns to SOAKING, not ACTIVE, because
status was never changed.

The evaluator filters by `status IN ('ACTIVE', 'SOAKING') AND killed_at
IS NULL`. A killed launch is excluded from evaluation — same effect as
ramp 0%, but without losing the ramp percentage or lifecycle state.
Killing and unkilling is instant and does not require a config change
or sync.

**Kill vs abandon:**

- **Kill** (`killed_at IS NOT NULL`) — reversible emergency disable.
  Lifecycle state and ramp percentage are preserved. Unkill clears
  `killed_at` and the launch resumes where it left off.
- **Abandon** (`status = ABANDONED`) — permanent. The launch is over and
  will not be landed. Irreversible. Used when the launch is no longer
  wanted. Stale launches without config definitions are also marked
  ABANDONED by the sync tool.

**Lifecycle:**

```
CREATED → ACTIVE → SOAKING → COMPLETED

          ABANDONED  (from any state except COMPLETED)

          killed_at  (orthogonal — can be set/cleared on any active state)
```

**CLI:**

```
pb launch kill <launch>      Emergency: disable launch, preserve state
pb launch unkill <launch>    Resume launch where it left off
pb launch abandon <launch>   Permanent: launch is over, not landing
pb kill <flag>               Kill entire flag (unchanged from today)
```

### Caching

The bounded-output principle from the evaluation-context design applies
unchanged. Each active launch adds a binary (in-ramp / out-of-ramp)
component to the cache key:

```
launch:<launch_id>=1|0
```

A flag with two active launches has ×4 cache cardinality (2 × 2),
regardless of the hash dimension's cardinality. The hash computation
(`FNV-32a(launch_id + \x00 + dim_value) % 100 < ramp_pct`) runs at
cache-key construction time, same as finite-filter membership checks.

The cache key builder receives launches as part of the flag state and
includes the in/out indicator for each. This is unchanged from the current
implementation in `condcache.go`.

### Database schema

The existing `feature_flags.launches` table (migration 007) needs the
following changes:

```sql
-- Remove columns that are no longer needed:
--   flag_id       (binding is now inline in YAML, not on the launch)
--   population_cel (targeting handled by condition when clauses)
--   value         (values are per-condition overrides, not per-launch)
--
-- The launch table becomes purely about the launch entity itself:
ALTER TABLE feature_flags.launches
  DROP COLUMN flag_id,
  DROP COLUMN population_cel,
  DROP COLUMN value;

-- Rename feature_id → scope_feature_id to clarify its purpose: where
-- the launch is defined, not what it affects. Make nullable for
-- cross-feature launches defined in launches/.
ALTER TABLE feature_flags.launches
  RENAME COLUMN feature_id TO scope_feature_id;
ALTER TABLE feature_flags.launches
  ALTER COLUMN scope_feature_id DROP NOT NULL;

-- Add affected_features: all features that reference this launch.
-- Populated by sync from config file references.
ALTER TABLE feature_flags.launches
  ADD COLUMN affected_features TEXT[] NOT NULL DEFAULT '{}';

-- Add description column:
ALTER TABLE feature_flags.launches
  ADD COLUMN description TEXT;

-- Drop the now-invalid index on flag_id:
DROP INDEX IF EXISTS feature_flags.idx_launches_flag;
```

The resulting table:

| Column | Type | Description |
|--------|------|-------------|
| `launch_id` | VARCHAR(255) PK | Unique identifier (matches YAML key or filename) |
| `scope_feature_id` | VARCHAR(255) FK, nullable | Where the launch is defined. Non-null = feature config, null = `launches/` directory. Used for UI treatment and scoping enforcement. |
| `affected_features` | TEXT[] NOT NULL | All features that reference this launch. Populated by sync. |
| `dimension` | VARCHAR(255) | Hashable dimension name |
| `ramp_percentage` | INT (0–100) | Current ramp, controlled via admin UI / CLI |
| `status` | VARCHAR(20) | CREATED, ACTIVE, SOAKING, COMPLETED, ABANDONED |
| `killed_at` | TIMESTAMPTZ, nullable | Non-null = launch is killed (reversible emergency disable). Orthogonal to status. |
| `description` | TEXT | Human-readable description |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | |

The condition overrides (which values change under which launch) are stored
in the existing `flags.conditions` JSONB column. The `StoredCondition`
struct gains `launch_id` and `launch_value` fields:

```go
type StoredCondition struct {
    CEL          *string         `json:"cel,omitempty"`
    Value        json.RawMessage `json:"value"`
    Comment      string          `json:"comment,omitempty"`
    LaunchID     string          `json:"launch_id,omitempty"`     // at most one
    LaunchValue  json.RawMessage `json:"launch_value,omitempty"`  // protojson-encoded FlagValue
}
```

### Sync pipeline changes

`SyncConditions` changes:

1. **Collect launch definitions** from two sources:
   - Feature config files: `launches:` section (feature-scoped)
   - Top-level `launches/` directory: standalone YAML files (cross-feature)

2. **Validate scope**: Feature-scoped launches are only referenced from
   their own feature. Cross-feature launches may be referenced from any
   feature.

3. **Compile condition overrides**: For each condition with a `launch:` key,
   validate that the referenced launch exists, the override value is
   type-compatible with the flag, and encode the override into the
   `StoredCondition.LaunchID`/`LaunchValue` fields.

4. **Collect affected features**: For each launch, collect the set of
   feature IDs whose configs reference it. Validate that feature-scoped
   launches have exactly one affected feature matching `scope_feature_id`.

5. **Upsert launch records**: Insert/update the `launches` table. Set
   `affected_features` on every sync (always reflects current config).
   `ramp_percentage` is only set on insert — existing ramps are preserved.

6. **Stale launch cleanup**: Launches in the database that are no longer
   defined in any config file are marked ABANDONED (not deleted — audit
   trail).

### Evaluator fetch path

With `flag_id` removed from the `launches` table, the evaluator can no
longer load launches via `WHERE flag_id = $1`. Instead, the fetch path
uses the launch IDs embedded in the flag's condition chain:

1. **Load flag state** (existing path): fetch `conditions` JSONB from the
   `flags` table. Parse `StoredCondition` entries.
2. **Extract launch IDs**: collect distinct `launch_id` values from the
   parsed conditions.
3. **Batch-fetch launch records**: query the `launches` table with
   `WHERE launch_id = ANY($1) AND status IN ('ACTIVE', 'SOAKING')
   AND killed_at IS NULL`. Returns launch metadata (dimension,
   ramp_percentage, status) needed for hash evaluation and cache key
   construction.
4. **Build `CachedLaunch` slice**: one entry per active launch, keyed by
   launch_id. The `CachedLaunch` struct no longer carries `Population`
   or `Value` — those are in the condition chain.
5. **Cache key construction**: unchanged — each active launch adds a
   binary in/out component via `HashBucket`.

The batch fetch in step 3 replaces the current per-flag
`fetchLaunches(ctx, flagID)` query. For flags with no launch overrides
(the common case), step 2 yields an empty set and step 3 is skipped.

**Never-throw contract:** Launch evaluation follows the same "log and
degrade" pattern as the existing condition evaluator:

- **Batch fetch fails** (DB error): log warning, evaluate condition chain
  without launch overrides. All entities see base condition values.
- **Launch ID referenced by condition is missing** (stale data, race
  condition): log warning, skip the launch override for that condition.
  Entity sees the base condition value.
- **`LaunchValue` cannot be decoded** (corrupted JSONB): log warning,
  skip the launch override. Entity sees the base condition value.
- **Hash dimension missing from context** (should not happen with scope
  enforcement, but defensive): hash the empty string. Deterministic
  bucket assignment, no panic.

In all cases, the evaluator returns a value and a source label. It never
returns an error or panics. This preserves the never-throw guarantee from
`docs/philosophy.md`.

### Mechanical landing: `pb launch land`

The most important CLI command. When a launch is SOAKING (100% ramp, soak
period complete), `pb launch land` transforms the config to make the launch
values permanent:

**Algorithm:**

1. Verify the launch status is SOAKING.
2. For each config file that references the launch:
   a. For each condition with a `launch:` override matching the launch ID:
      - Replace the condition's `value` with the launch override value.
      - Remove the `launch:` key entirely.
   b. For each static flag with a `launch:` override matching the launch ID:
      - Replace `value` with the launch override value.
      - Remove the `launch:` key.
3. Delete the launch definition:
   - If feature-scoped: remove the entry from the `launches:` section.
     If the `launches:` section is now empty, remove it.
   - If cross-feature: delete `launches/<launch>.yaml`.
4. Commit the changes and open a PR via `gh`.

**Example transform:**

Before:
```yaml
launches:
  daily-digest-rollout:
    dimension: user_id
    ramp_percentage: 25

flags:
  digest_frequency:
    conditions:
      - when: 'ctx.plan == PlanLevel.PRO'
        value: "weekly"
        launch:
          id: daily-digest-rollout
          value: "daily"
      - otherwise: "weekly"
```

After `pb launch land daily-digest-rollout`:
```yaml
flags:
  digest_frequency:
    conditions:
      - when: 'ctx.plan == PlanLevel.PRO'
        value: "daily"
      - otherwise: "weekly"
```

For Phase 1 launches (per-condition value overrides), the transform is
purely mechanical — YAML key replacement and deletion. No CEL parsing, no
semantic analysis. Phase 2 extends `pb launch land` with CEL
simplification for `launch.in_ramp()` expressions (see "Phase 2" below).

**Correctness invariant — dimension presence and hashing:** Launch
dimensions must be present in all scopes of all affected features (see
"Dimension presence" below). Launch hashing includes all dimension values,
including empty strings — the current `dimValue != ""` guard in
`EvaluateLaunches` is removed (see "Launch hashing" below). At 100% ramp,
`hash % 100 < 100` holds for every entity regardless of dimension value.
Landing — which promotes the launch value to the base value — produces
identical evaluations. No edge cases.

**PR generation:** `pb launch land` creates a branch, commits the
transformed config, and opens a PR. The PR description lists every flag
and condition that changed. The PR is reviewed like any config change —
the diff is small and obvious because only values change.

### Dimension presence

Dimensions are classified as **required** or **optional** via a new
`presence` field on `DimensionOptions`:

```protobuf
enum DimensionPresence {
  DIMENSION_PRESENCE_UNSPECIFIED = 0;
  DIMENSION_PRESENCE_REQUIRED = 1;  // Always present. Consumer must set it.
  DIMENSION_PRESENCE_OPTIONAL = 2;  // May be absent (e.g., user_id when logged out).
}

enum DimensionDistribution {
  DIMENSION_DISTRIBUTION_UNSPECIFIED = 0;
  DIMENSION_DISTRIBUTION_UNIFORM = 1;     // High cardinality, stable values. Suitable for percentage-based hashing.
  DIMENSION_DISTRIBUTION_CATEGORICAL = 2; // Low/bounded cardinality. Enum-like. Not suitable for hashing.
}

message DimensionOptions {
  string description = 1;
  DimensionDistribution distribution = 2; // replaces hashable (bool, field 2) and bounded (bool, field 3)
  DimensionPresence presence = 4;
  // Fields 2 and 3 were previously hashable (bool) and bounded (bool).
  // The distribution enum subsumes both. Wire-compatible: old bool true (1)
  // maps to UNIFORM (1), so existing descriptors with hashable: true decode
  // as distribution: UNIFORM without recompilation.
}
```

The `distribution` enum replaces two booleans:

| Old | New | Cache key strategy |
|-----|-----|--------------------|
| `hashable: true` | `UNIFORM` | Hash-based bounded projection (×2 per launch) |
| `bounded: true` | `CATEGORICAL` | Raw value in cache key (bounded by value space) |
| neither | `UNSPECIFIED` | Raw value in cache key (unbounded, sync warns, LRU cap) |

Enum and bool proto field types are inherently categorical — inferred from
the proto type, no annotation needed (same as today where
`BoundedDimsFromDescriptor` auto-marks them). The `distribution` field is
only needed for string and int64 dimensions.

Example context definition:

```protobuf
message EvaluationContext {
  option (pbflags.context) = {};

  // Required, uniform — always present, high cardinality.
  // Anonymous stable ID for logged-out users, user_id for logged-in.
  string session_id = 1 [(pbflags.dimension) = {
    description: "Stable session identifier"
    distribution: DIMENSION_DISTRIBUTION_UNIFORM
    presence: DIMENSION_PRESENCE_REQUIRED
  }];

  // Optional, uniform — only present when authenticated.
  string user_id = 2 [(pbflags.dimension) = {
    description: "Authenticated user identifier"
    distribution: DIMENSION_DISTRIBUTION_UNIFORM
    presence: DIMENSION_PRESENCE_OPTIONAL
  }];

  // Optional, categorical (inferred from enum type — annotation optional).
  PlanLevel plan = 3 [(pbflags.dimension) = {
    description: "User plan tier"
    presence: DIMENSION_PRESENCE_OPTIONAL
  }];
}
```

**Launch dimensions must be uniform and present in all affected scopes.**
The launch dimension's job is traffic splitting, not targeting. Targeting
is the condition chain's job. The dimension must be:

- `distribution: UNIFORM` — high cardinality for smooth hash distribution
  across the 0–100% ramp range.
- Present in every scope across all affected features. This ensures the
  dimension is set whenever any affected feature is evaluated, so 100%
  ramp truly means 100% and landing is behavior-preserving.

The sync tool enforces both:

```
error: launch "pro-v2" hashes on dimension "tenant_id" but feature
  "user_prefs" is available in scope "user" which does not provide
  "tenant_id". Use a dimension available in all scopes (session_id,
  user_id) or remove user_prefs flags from the launch.
```

```
error: launch "region-test" hashes on dimension "region" which has
  distribution: CATEGORICAL. Launch dimensions must be UNIFORM.
  Categorical dimensions have too few values for smooth ramp control.
```

**Consumer guidance for user-stable hashing:** If launches should be
stable across a user's sessions, the consumer sets `session_id = user_id`
when the user is authenticated. When anonymous, the consumer sets
`session_id` to a stable device or installation identifier. The launch
hashes on `session_id` regardless — it's always present, and for
logged-in users it produces the same bucket as hashing on `user_id`
directly would.

**Codegen enforcement via evaluation scopes:** The codegen produces
typed feature accessor objects per scope that make missing dimensions a
compile error, not a runtime surprise.

Scopes define which dimensions are available in a given execution context.
Globally required dimensions (e.g., `session_id`) are implicit in every
scope — they do not need to be listed. The `dimensions` field lists only
the additional dimensions the scope provides beyond the global set.
Features declare which scopes they're available in:

```protobuf
// Scope definitions — globally required dims (session_id) are implicit
option (pbflags.scope) = { name: "anon" };
option (pbflags.scope) = { name: "user", dimensions: ["user_id"] };
option (pbflags.scope) = { name: "tenant", dimensions: ["user_id", "tenant_id"] };

// Features reference scopes directly
message BillingFlags {
  option (pbflags.feature) = { id: "billing", scopes: ["tenant"] };
}
message UserPrefsFlags {
  option (pbflags.feature) = { id: "user_prefs", scopes: ["user", "tenant"] };
}
message SiteFlags {
  option (pbflags.feature) = { id: "site_wide", scopes: ["anon", "user", "tenant"] };
}
```

Codegen is direct — a scope gets a feature's accessor if the feature
lists that scope:

| Generated type | Feature accessors |
|----------------|-------------------|
| `AnonFeatures` | `SiteWide()` |
| `UserFeatures` | `SiteWide()`, `UserPrefs()` |
| `TenantFeatures` | `SiteWide()`, `UserPrefs()`, `Billing()` |

**Generated Go code:**

```go
// Per-scope feature accessors — constructors require scope dimensions
type AnonFeatures struct { /* ... */ }
func NewAnonFeatures(eval Evaluator, sessionID string) *AnonFeatures
func (f *AnonFeatures) SiteWide() *SiteFlags

type UserFeatures struct { /* ... */ }
func NewUserFeatures(eval Evaluator, sessionID, userID string) *UserFeatures
func (f *UserFeatures) SiteWide() *SiteFlags
func (f *UserFeatures) UserPrefs() *UserPrefsFlags

type TenantFeatures struct { /* ... */ }
func NewTenantFeatures(eval Evaluator, sessionID, userID, tenantID string) *TenantFeatures
func (f *TenantFeatures) SiteWide() *SiteFlags
func (f *TenantFeatures) UserPrefs() *UserPrefsFlags
func (f *TenantFeatures) Billing() *BillingFlags

// Duck-typed interfaces — generated per feature
type HasBilling interface { Billing() *BillingFlags }
type HasUserPrefs interface { UserPrefs() *UserPrefsFlags }
type HasSiteWide interface { SiteWide() *SiteFlags }
```

**Call site:**

```go
tenantFeatures.Billing().DigestFrequency()
userFeatures.UserPrefs().Theme()
anonFeatures.SiteWide().DarkMode()

// anonFeatures.Billing() → compile error: no such method
```

**Middleware creates the features object once per request:**

```go
func TenantMiddleware(eval Evaluator) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            features := dims.NewTenantFeatures(eval, sessionID, userID, tenantID)
            ctx := dims.ContextWith(r.Context(), features)
            next.ServeHTTP(w, r.WithContext(ctx))
        })
    }
}
```

**Duck typing for handlers** that don't need to know the concrete scope:

```go
func handleInvoice(features HasBilling) {
    freq := features.Billing().DigestFrequency()
}
// Accepts TenantFeatures or any future scope with Billing()
```

The compile-time guarantees:
- Scope constructors require their dimensions — can't create a
  `TenantFeatures` without `tenantID`.
- Feature accessor methods only exist on scopes the feature lists —
  can't call `Billing()` on `AnonFeatures`.
- Duck-typed interfaces let handlers declare what they need without
  coupling to a specific scope.

**Launch dimension validation** uses scopes: the launch dimension must
be in the dimensions of every scope across all affected features. For a
launch affecting billing (`scopes: ["tenant"]`) and user_prefs
(`scopes: ["user", "tenant"]`), the valid dimensions are the intersection
of all referenced scopes' dimensions — `[session_id, user_id]`. The
launch can hash on `session_id` or `user_id` but not `tenant_id` (because
user_prefs is available in the "user" scope which lacks `tenant_id`).

This applies to both Phase 1 and Phase 2 launches.

### Proto changes

Update `EvaluationSource` to clarify the launch source:

```protobuf
EVALUATION_SOURCE_LAUNCH = 9;  // Matched a launch override on a condition
```

The `CompiledLaunch` message in `bundle.proto` is simplified — it no longer
carries `flag_id`, `population_cel`, or `value_json` (those are now in the
condition chain):

```protobuf
message CompiledLaunch {
  string launch_id = 1;
  string dimension = 3;
  int32 ramp_percentage = 6;
  string scope_feature_id = 7;           // nullable — empty for cross-feature
  repeated string affected_features = 8;
  string description = 9;
}
```

Cross-feature launches are not naturally owned by a `CompiledFeature`.
The bundle proto gains a top-level launches field:

```protobuf
message CompiledBundle {
  repeated CompiledFeature features = 1;
  string cel_version = 2;              // existing field — do not reuse
  repeated CompiledLaunch launches = 3; // all defined launches from config
}
```

The bundle is a config compilation artifact — it contains all launch
definitions from config, not filtered by runtime status. Active/soaking/unkilled
filtering happens at the evaluator: when upserting from the bundle,
`ramp_percentage` is only applied on insert (preserving operator-set
ramp), and the evaluator's fetch path filters by
`status IN ('ACTIVE', 'SOAKING')`.

Each `CompiledFeature` continues to carry its conditions (which now include
`launch_id`/`launch_value` on individual entries). The evaluator joins
these at load time: parse conditions, extract launch IDs, look up the
corresponding `CompiledLaunch` entries for ramp percentage and dimension.

### Validation rules

The sync tool enforces:

1. **Launch exists**: Every `launch:` reference on a condition must resolve
   to a defined launch (in the same feature config or in `launches/`).
2. **Single override per condition**: A condition (or static value) may
   have at most one `launch:` override. Multiple overlapping launches
   require Phase 2 `launch.in_ramp()` in CEL.
3. **Dimension is uniform and present in all affected scopes**: The launch
   dimension must have `distribution: UNIFORM` and must appear in the
   dimensions of every scope across all affected features. See "Dimension
   presence" above.
4. **Scope enforcement**: Feature-scoped launches cannot be referenced from
   other features.
5. **Type compatibility**: Launch override values must match the flag's
   proto type.
6. **No duplicate launch IDs**: A launch ID cannot be defined in both a
   feature config and `launches/`, or in multiple feature configs.
7. **`ramp_percentage` range**: 0–100.
8. **Condition-scope compatibility**: Every CEL dimension referenced by a
   feature's conditions must be available in every scope the feature
   declares. See "Evolution and compatibility" below.
9. **Active launch immutability**: Launches in ACTIVE or SOAKING status
   (including killed) cannot change `dimension` or ownership. See
   "Evolution and compatibility" below.

### Evolution and compatibility

Scopes, features, and active launches form a stable API surface. Changes
to them can break consumers, invalidate launches, or expose flags to
contexts they weren't written for. The following rules govern evolution.

**Scopes are stable API.** Changing a scope's dimension set is a breaking
change — adding a dimension changes constructor signatures, removing one
can invalidate active launches and generated methods. Prefer adding a new
scope over mutating an existing one. `pbflags-lint` flags scope dimension
changes when comparing against a previous descriptor baseline:

```
error: scope "user" dimension set changed (added: "org_id").
  This changes the UserFeatures constructor signature. Either add a
  new scope (e.g., "org_user") or coordinate the change with consumers.
```

**Feature scope changes require condition compatibility.** Adding a
feature to a broader scope exposes it to contexts where some dimensions
may be absent. The sync tool validates that every CEL dimension referenced
by a feature's conditions is available in every scope the feature declares:

```
error: feature "user_prefs" added to scope "anon", but condition
  'ctx.user_id != ""' on flag "theme" references dimension "user_id"
  which is not in scope "anon". Either remove "anon" from
  user_prefs.scopes or rewrite the condition.
```

This is separate from launch dimension validation — it applies to all
conditions, not just launch overrides.

**Active launches are immutable in bucketing-affecting fields.** For a
launch in ACTIVE or SOAKING status (including killed launches — `killed_at`
does not relax this), the sync tool rejects changes to `dimension`
(rebuckets every entity mid-launch) and ownership
(feature-scoped ↔ cross-feature). To change these, land or abandon the
launch first and create a new one:

```
error: launch "pro-v2" is ACTIVE but dimension changed from
  "session_id" to "user_id". Land or abandon the launch before
  changing bucketing fields, or create a new launch ID.
```

**Generated API removals surfaced at lint time.** Moving a feature out
of a scope removes generated accessor methods. Renaming or removing a
scope removes generated `*Features` types. These are useful compile
errors, but the config author should see them before they break consumers.
`pbflags-lint --breaking-check <old-descriptor>` reports:

- Removed scope types (e.g., `UserFeatures` deleted)
- Removed feature accessors (e.g., `UserFeatures.UserPrefs()` removed)
- Changed constructor signatures (e.g., new required dimension parameter)

This is analogous to `buf breaking` for proto definitions — a CI gate
that catches generated API regressions before they reach consumers.

## Phase 2: Launch state in CEL

Phase 1 (above) covers per-condition value overrides — the common case
where a launch changes what a condition returns. Phase 2 adds the ability
to use launch ramp state directly in CEL expressions, enabling structural
changes to the condition chain under a launch.

### Motivation

Some launches need to do more than override a value — they need to add new
conditions or change the structure of evaluation:

```yaml
# "Under the launch, add a new EU-specific price tier that doesn't
#  exist in the base config."
flags:
  monthly_price:
    conditions:
      - when: 'launch.in_ramp("eu-pricing") && ctx.region == Region.EU'
        value: 24.99
      - when: 'ctx.plan == PlanLevel.PRO'
        value: 29.99
      - otherwise: 9.99
```

This can't be expressed with per-condition value overrides because it
introduces a new condition that doesn't exist in the base chain.

### CEL integration

Add a `launch` variable to the CEL environment:

```
launch.in_ramp(launch_id: string) → bool
```

Returns true if the current entity is in the named launch's ramp (i.e.,
`hash(launch_id + dim_value) % 100 < ramp_pct`). The launch must be
ACTIVE or SOAKING.

The function is pure and deterministic — same entity always gets the same
result for a given ramp percentage. This preserves the condition chain's
"pure function of context" property with one extension: the evaluation
also depends on operational state (the launch ramp percentage in the DB).

### Caching impact

`launch.in_ramp("X")` in a CEL expression is equivalent to the Phase 1
cache key component: a binary in/out projection. The dimension classifier
recognizes `launch.in_ramp(...)` calls and classifies them as bounded (×2
cardinality per launch reference).

### Mechanical landing with CEL simplification

Landing a Phase 2 launch requires CEL-level transformation, not just YAML
key replacement. At 100% ramp, `launch.in_ramp("X")` is always true.
Launch dimensions must be present in all scopes of all affected features
(see "Dimension presence" above), so the hash dimension is always set when
the feature is evaluated and `hash % 100 < 100` holds for every entity.
Substituting `true` is unconditionally behavior-preserving.

The landing algorithm:

1. **Substitute**: Replace `launch.in_ramp("<launch>")` with `true` in
   every CEL expression that references the launch.
2. **Fold**: Run the `cel-go` `StaticOptimizer` with
   `NewConstantFoldingOptimizer` to simplify expressions:
   - `true && ctx.region == Region.EU` → `ctx.region == Region.EU`
   - `true || X` → `true`
   - `false && X` → `false`
3. **Prune**: Remove conditions whose expressions folded to `false`
   (they can never match). Merge conditions whose expressions folded
   to `true` with earlier conditions (they always match, shadowing
   everything below).
4. **Unparse**: Convert simplified ASTs back to CEL strings via
   `cel.AstToString()`.
5. **Write back**: Update the YAML config with simplified expressions.
   Remove the launch definition.

**Implementation uses first-party cel-go tooling** (pseudocode — actual
API signatures may differ slightly across cel-go versions; verify against
the version in `go.mod` before implementing):

```go
// 1. Custom ASTOptimizer: replace launch.in_ramp("X") → true
type launchSubstituter struct {
    launchID string
    result   bool // true for landing, false for abandoning
}

func (s *launchSubstituter) Optimize(ctx *cel.OptimizerContext, a *ast.AST) *ast.AST {
    ast.MatchDescendants(a, func(node ast.NavigableExpr) bool {
        if node.Kind() == ast.CallKind &&
           node.AsCall().FunctionName() == "in_ramp" {
            args := node.AsCall().Args()
            if len(args) == 1 && args[0].Kind() == ast.LiteralKind {
                if args[0].AsLiteral().Value().(string) == s.launchID {
                    ctx.UpdateExpr(node, ctx.NewLiteral(types.Bool(s.result)))
                }
            }
        }
        return false
    })
    return a
}

// 2. Chain with constant folder
folder, _ := cel.NewConstantFoldingOptimizer()
opt, _ := cel.NewStaticOptimizer(&launchSubstituter{launchID: "eu-pricing", result: true}, folder)
optimized, _ := opt.Optimize(env, checkedAST)
simplified := cel.AstToString(optimized)
```

**Example transform:**

Before:
```yaml
conditions:
  - when: 'launch.in_ramp("eu-pricing") && ctx.region == Region.EU'
    value: 24.99
  - when: 'ctx.plan == PlanLevel.PRO'
    value: 29.99
  - otherwise: 9.99
```

After `pb launch land eu-pricing`:
```yaml
conditions:
  - when: 'ctx.region == Region.EU'
    value: 24.99
  - when: 'ctx.plan == PlanLevel.PRO'
    value: 29.99
  - otherwise: 9.99
```

The `launch.in_ramp("eu-pricing")` was replaced with `true`, the folder
simplified `true && ctx.region == Region.EU` to `ctx.region == Region.EU`,
and the result was unparsed back to a clean expression.

### Phase 2 validation rules

The sync tool adds:

1. **`launch.in_ramp()` references resolve**: Every `launch.in_ramp("X")`
   call must reference a defined launch.
2. **Population dimension restriction**: CEL expressions containing
   `launch.in_ramp()` may reference unbounded dimensions freely (the
   launch hash itself is the bounded projection). However, `launch.in_ramp()`
   should appear in conjunction (`&&`) with other predicates, not as a
   standalone condition — a standalone `launch.in_ramp("X")` is just a
   Phase 1 override and should use the simpler syntax.

### Interaction between Phase 1 and Phase 2

A single launch can use both mechanisms:

```yaml
flags:
  monthly_price:
    conditions:
      # Phase 2: new condition that only exists under the launch
      - when: 'launch.in_ramp("pro-v2") && ctx.region == Region.EU'
        value: 24.99
      # Phase 1: existing condition with value override
      - when: 'ctx.plan == PlanLevel.PRO'
        value: 29.99
        launch:
          id: pro-v2
          value: 19.99
      - otherwise: 9.99
```

The landing transform handles both:
- Phase 1 overrides: YAML key replacement (as described above)
- Phase 2 expressions: CEL substitution + folding

Both transforms run in the same `pb launch land` invocation.

## Migration from current implementation

The current launch implementation (migration 007, `launch.go`,
`configfile.go`, `sync/conditions.go`) stores a flat value per launch
and evaluates launches as a separate step before conditions.

**This is a breaking schema change.** The migration drops columns that
the current evaluator and admin code query (`flag_id`, `population_cel`,
`value`). All existing launches must be completed or abandoned before
upgrading, and all services must be upgraded together. This is acceptable
because the launch feature is pre-release — no production launches exist
against the old schema.

If launches were in production use, the migration would need to be staged
across releases (add new columns → dual-read → backfill → drop old
columns), per the repo's backwards-compatible migration rules in
`docs/contributing.md`. That staged approach is not warranted here.

**Schema migration:**

1. Verify no launches exist with status IN ('ACTIVE', 'SOAKING').
2. Drop `flag_id`, `population_cel`, `value` columns.
3. Rename `feature_id` → `scope_feature_id`, drop NOT NULL.
4. Add `affected_features` (TEXT[]), `description` (TEXT),
   `killed_at` (TIMESTAMPTZ, nullable).

**Code migration:**

5. **Config format**: The YAML `launches:` section changes from
   `{flag, dimension, population, value, ramp_percentage}` to
   `{dimension, ramp_percentage, description}`. Launch-to-flag binding
   moves inline to conditions.
6. **Sync pipeline**: `SyncConditions` encodes launch overrides into
   `StoredCondition.LaunchID`/`LaunchValue` instead of the `launches`
   table's `value` column.
7. **Evaluator**: Launch evaluation moves from a separate
   `EvaluateLaunches()` step to inline within condition evaluation.
   The `CachedLaunch` struct drops `Population` and `Value` fields.
8. **Admin store**: Query changes to reflect the simplified launch table.
   Add `KillLaunch`/`UnkillLaunch` methods (set/clear `killed_at`).
9. **Admin UI**: Conditions table gains a "Launch overrides" column.
   Separate launches table simplified. Kill/unkill controls added.
10. **Bundle proto**: Reserve `CompiledFeature.launches` (field 6) — this
    field currently carries per-feature launches in the old model.
    Launches move to `CompiledBundle.launches` (field 3). During the
    transition, the loader ignores `CompiledFeature.launches` and reads
    only from the bundle-level field.

**Proto migration:**

11. **DimensionOptions proto**: Replace `hashable` (bool, field 2) and
   `bounded` (bool, field 3) with `distribution` (enum, field 2) and
   add `presence` (enum, field 4). Wire-compatible: old `hashable: true`
   encodes as varint 1 on field 2, which decodes as
   `DIMENSION_DISTRIBUTION_UNIFORM` (enum value 1). Old `bounded: true`
   on field 3 is ignored (field 3 is no longer declared).
   **Cache impact during migration**: string/int64 dimensions that relied
   on `bounded: true` for cache classification will temporarily degrade
   from bounded raw-key behavior to unbounded/LRU behavior until
   consumers regenerate descriptors with `distribution: CATEGORICAL`.
   Enum and bool dimensions are unaffected (inherently categorical).
   Consumers should recompile promptly after upgrading.
12. **FeatureOptions proto**: Add `scopes` (repeated string) to
    `FeatureOptions`. Existing features without it need scopes added.
13. **Scope definitions**: Add `pbflags.scope` file-level option. Consumers
    define 2–3 scopes matching their execution contexts (anon, user,
    tenant, etc.).
14. **Codegen**: Replace the current single evaluator codegen with
    per-scope `*Features` types. Each scope gets accessor methods for
    features that list it in their `scopes`. Generate duck-typed interfaces
    per feature (`HasBilling`, etc.). Existing consumers need to update
    from `feature.New(evaluator)` to the scope-based pattern.

## Future work

### Approval gates

Launches gain an optional approval workflow:

```yaml
# launches/pro-v2.yaml
dimension: session_id
ramp_percentage: 0
approval:
  required: true
  approvers:
    - product-team
    - eng-leads
```

`pb launch ramp` requires approval before the ramp can be increased.
The approval state is tracked in the database and surfaced in the admin UI.

### Staged ramp plans

A launch can define a ramp schedule:

```yaml
ramp_plan:
  - percentage: 1
    soak: 24h
  - percentage: 10
    soak: 48h
  - percentage: 50
    soak: 72h
  - percentage: 100
```

The operator advances the ramp manually via `pb launch ramp`, using the
plan as a checklist. Automatic advancement (scheduled jobs, health-metric
gating) is a later feature.

### Experiments

Experiments build on the launch primitive but assign entities to named
variants instead of binary in/out ramp:

```yaml
experiments:
  notification-cadence:
    dimension: user_id
    variants:
      control:
        percentage: 50
      daily:
        percentage: 25
      realtime:
        percentage: 25
```

Variant assignment uses the same hash mechanism (`hash(experiment_id +
dim_value) % 100`) but maps to variant buckets instead of a single ramp
threshold. The CEL integration (Phase 2) naturally extends:
`experiment.variant("notification-cadence")` returns the variant name.

Experiment logging is handled by the generated client SDK via an optional
`ExperimentLogger` callback — the evaluator remains stateless.
