# Launches

**Status:** Draft
**Date:** 2026-04-14
**Author:** bmt
**Builds on:** [Evaluation Context and Conditional Flags](2026-04-12-evaluation-context-and-conditions.md)

## Problem

pbflags has conditions (CEL-based value selection) and static values, but no
mechanism for gradually rolling out a value change. Today, changing a flag value
means editing the YAML config and syncing — the change takes effect for 100%
of traffic immediately. There's no way to say "roll this change out to 10% of
users, observe, then increase."

The existing launch implementation (migration 007, `evaluator/launch.go`)
stores a flat value per launch and evaluates it independently of the condition
chain. This design replaces that model with one that integrates launches into
the condition chain itself, supports multi-feature coordination, and provides
a mechanical path from "ramping" to "landed."

## Terminology

**Launch** — the unit of gradual rollout. A launch has a dimension (for
hashing), a ramp percentage (for traffic splitting), and a lifecycle
(CREATED → ACTIVE → BAKED → COMPLETED / ABANDONED). A launch may affect
one flag (feature-scoped) or many flags across features (cross-feature).

The word "launch" was chosen over "rollout" because launches are more than
a mechanical ramp — they're the organizational unit for shipping changes.
Future work adds approval gates, staged ramp plans, and review workflows
to the launch concept. See "Future work" at the end of this document.

## Goals

1. **Per-condition value overrides** — a launch changes what specific
   conditions return, not the structure of the condition chain.
2. **Multi-feature launches** — a single launch can affect flags across
   multiple features, ramping them together.
3. **Inline visibility** — reading a feature's config shows exactly how
   a launch affects it. No indirection to a separate launch definition
   to understand behavior.
4. **Mechanical landing** — `pb launch land <launch>` transforms the YAML
   config to promote launch values to defaults with guaranteed correctness.
   The transform is a pure YAML operation — no CEL reasoning required.
5. **Bounded caching** — launches do not add unbounded cardinality to
   the condition cache. The existing bounded-output principle applies.

## Non-Goals

- **Condition chain replacement** (Option A from design discussion).
  Replacing the full chain is redundant and creates silent divergence risk
  when the base chain is edited during a launch. Per-condition overrides
  avoid this class of bug.
- **Launch state in CEL** (Option C). Deferred to Phase 2. Requires a
  CEL simplifier for mechanical landing. See "Phase 2" below.
- **Approval gates and staged ramp plans.** Future work on the launch
  lifecycle. The primitive ships first.
- **Experiments** (A/B testing with variant assignment and logging).
  Orthogonal; depends on launch infrastructure but has its own design
  surface.

## Design

### YAML config format

#### Feature-scoped launch (single feature)

When a launch affects only one feature, define it in the feature's own
config file:

```yaml
# features/notifications/notifications.yaml
feature: notifications

launches:
  daily-digest-rollout:
    dimension: user_id
    ramp_percentage: 25

flags:
  digest_frequency:
    conditions:
      - when: 'ctx.plan == PlanLevel.ENTERPRISE'
        value: "daily"
      - when: 'ctx.plan == PlanLevel.PRO'
        value: "weekly"
        launch:
          daily-digest-rollout: "daily"
      - otherwise: "weekly"
```

The `launches:` section defines the launch — its hash dimension and initial
ramp percentage. The `launch:` key on individual conditions defines what
value that condition returns when the entity is in the launch's ramp.

In this example: Pro users in the 25% ramp get `"daily"` instead of
`"weekly"`. Enterprise users are unaffected (their condition has no launch
override). Users outside the ramp evaluate the condition chain normally.

**Scoping rule:** A launch defined in a feature config file can only be
referenced by conditions in that same file. The sync tool errors if a
feature-scoped launch is referenced from another feature:

```
error: flag "billing/3" references launch "daily-digest-rollout" which is
  scoped to feature "notifications". Move the launch to launches/ to use
  it across features.
```

#### Cross-feature launch (multiple features)

When a launch affects multiple features, define it in a standalone file
under a top-level `launches/` directory:

```
features/
  notifications/
    notifications.proto
    notifications.yaml
  billing/
    billing.proto
    billing.yaml
launches/
  pro-v2.yaml
```

```yaml
# launches/pro-v2.yaml
dimension: user_id
ramp_percentage: 10
```

The launch ID is the filename sans extension: `pro-v2.yaml` defines launch
`pro-v2`. No `launch_id:` field inside the file.

Referenced from multiple feature configs:

```yaml
# features/notifications/notifications.yaml
feature: notifications

flags:
  digest_frequency:
    conditions:
      - when: 'ctx.plan == PlanLevel.PRO'
        value: "weekly"
        launch:
          pro-v2: "daily"
      - otherwise: "weekly"
```

```yaml
# features/billing/billing.yaml
feature: billing

flags:
  monthly_price:
    conditions:
      - when: 'ctx.plan == PlanLevel.PRO'
        value: 29.99
        launch:
          pro-v2: 19.99
      - otherwise: 9.99
```

Both flags ramp together because they share the same launch. `pb launch ramp
pro-v2 50` moves both to 50%.

#### Static value overrides

Launches can also override static (non-conditional) flag values:

```yaml
flags:
  email_enabled:
    value: false
    launch:
      pro-v2: true
```

This is sugar for "the default value changes under the launch." The
mechanical landing transform replaces `value: false` with `value: true`
and removes the `launch:` key.

#### Multiple concurrent launches on one condition

A condition may have overrides from multiple launches:

```yaml
flags:
  digest_frequency:
    conditions:
      - when: 'ctx.plan == PlanLevel.PRO'
        value: "weekly"
        launch:
          pro-v2: "daily"
          digest-experiment: "realtime"
      - otherwise: "weekly"
```

**Evaluation order:** Launches are evaluated in the order they appear in
the `launch:` map. First matching launch wins. If the entity is in both
ramps, `pro-v2` takes precedence because it's listed first. YAML maps
preserve insertion order in the `gopkg.in/yaml.v3` decoder.

**Recommendation:** The sync tool warns when a condition has multiple
launch overrides, since ordering sensitivity is subtle. For most use cases,
multiple launches should target different conditions.

### Launch definition fields

| Field | Required | Description |
|-------|----------|-------------|
| `dimension` | Yes | Hashable dimension for ramp assignment. Must have `hashable: true` in proto `DimensionOptions`. |
| `ramp_percentage` | No | Initial ramp (0–100). Only applied on first sync (insert). Subsequent syncs preserve the operator-set ramp in the database. Default: 0. |
| `description` | No | Human-readable description for admin UI and CLI. |

Note the absence of `population` and `flag`. These fields from the current
implementation are unnecessary in this design:

- **`population`** is unnecessary because the condition's `when` clause
  already gates who sees the value. The launch doesn't need its own
  population filter — the condition chain handles targeting.
- **`flag`** is unnecessary because the launch-to-flag binding is expressed
  inline via the `launch:` key on individual conditions, not on the launch
  definition.

### Evaluation semantics

The evaluation precedence chain becomes:

```
1. Kill check (unchanged)
2. Condition chain evaluation (with launch overrides)
3. Static value / compiled default (unchanged)
```

Launches are no longer a separate evaluation step before conditions. Instead,
they modify what individual conditions return:

```
for each condition in chain:
    if condition.when matches context (or is "otherwise"):
        if condition has launch overrides:
            for each launch override (in order):
                if launch is ACTIVE or BAKED:
                    if entity is in launch ramp (hash check):
                        return launch override value  → source: LAUNCH
        return condition.value  → source: CONDITION
return compiled default  → source: DEFAULT
```

This preserves the "first condition match wins" invariant. The launch
override only applies within a matched condition — it doesn't change which
condition matches.

### Kill semantics

Kill behavior follows the hierarchy from the evaluation-context design doc:

| Kill target | Effect |
|-------------|--------|
| Flag | All evaluations return compiled default. Launches are bypassed. |
| Launch | Launch is deactivated (status → ABANDONED or admin kill). Entities in the ramp fall back to the condition's base value. Other launches on the flag are unaffected. |

Killing a launch is the common operational action. Killing a flag is the
nuclear option.

**Admin UI / CLI:** `pb launch kill <launch>` sets the launch status to
ABANDONED. `pb kill <flag>` kills the entire flag (unchanged from today).

### Caching

The bounded-output principle from the evaluation-context design applies
unchanged. Each active launch adds a binary (in-ramp / out-of-ramp)
component to the cache key:

```
launch:<launch_id>=1|0
```

A flag with two active launches has ×4 cache cardinality (2 × 2),
regardless of the hash dimension's cardinality. The hash computation
(`FNV-32a(launch_id + \x00 + dim_value) % 100 < ramp_pct`) runs at
cache-key construction time, same as finite-filter membership checks.

The cache key builder receives launches as part of the flag state and
includes the in/out indicator for each. This is unchanged from the current
implementation in `condcache.go`.

### Database schema

The existing `feature_flags.launches` table (migration 007) needs the
following changes:

```sql
-- Remove columns that are no longer needed:
--   flag_id       (binding is now inline in YAML, not on the launch)
--   population_cel (targeting handled by condition when clauses)
--   value         (values are per-condition overrides, not per-launch)
--
-- The launch table becomes purely about the launch entity itself:
ALTER TABLE feature_flags.launches
  DROP COLUMN flag_id,
  DROP COLUMN population_cel,
  DROP COLUMN value;

-- Add description column:
ALTER TABLE feature_flags.launches
  ADD COLUMN description TEXT;

-- Drop the now-invalid index on flag_id:
DROP INDEX IF EXISTS feature_flags.idx_launches_flag;
```

The resulting table:

| Column | Type | Description |
|--------|------|-------------|
| `launch_id` | VARCHAR(255) PK | Unique identifier (matches YAML key or filename) |
| `feature_id` | VARCHAR(255) FK | Feature that owns this launch (NULL for cross-feature) |
| `dimension` | VARCHAR(255) | Hashable dimension name |
| `ramp_percentage` | INT (0–100) | Current ramp, controlled via admin UI / CLI |
| `status` | VARCHAR(20) | CREATED, ACTIVE, BAKED, COMPLETED, ABANDONED |
| `description` | TEXT | Human-readable description |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | |

The condition overrides (which values change under which launch) are stored
in the existing `flags.conditions` JSONB column. The `StoredCondition`
struct gains a `launches` field:

```go
type StoredCondition struct {
    CEL     *string         `json:"cel,omitempty"`
    Value   json.RawMessage `json:"value"`
    Comment string          `json:"comment,omitempty"`
    // Launch overrides: launch_id → protojson-encoded FlagValue.
    Launches map[string]json.RawMessage `json:"launches,omitempty"`
}
```

### Sync pipeline changes

`SyncConditions` changes:

1. **Collect launch definitions** from two sources:
   - Feature config files: `launches:` section (feature-scoped)
   - Top-level `launches/` directory: standalone YAML files (cross-feature)

2. **Validate scope**: Feature-scoped launches are only referenced from
   their own feature. Cross-feature launches may be referenced from any
   feature.

3. **Compile condition overrides**: For each condition with a `launch:` key,
   validate that the referenced launch exists, the override value is
   type-compatible with the flag, and encode the override into the
   `StoredCondition.Launches` map.

4. **Upsert launch records**: Insert/update the `launches` table. As today,
   `ramp_percentage` is only set on insert — existing ramps are preserved.

5. **Stale launch cleanup**: Launches in the database that are no longer
   defined in any config file are marked ABANDONED (not deleted — audit
   trail).

### Admin store and UI changes

**Store methods:**

- `GetLaunchesForFlag` → replaced by `GetLaunch(launchID)` and
  `ListLaunches(featureID)` (launches are no longer per-flag).
- `UpdateLaunchRamp` — unchanged (operates on launch_id).
- `UpdateLaunchStatus` — unchanged.

**Admin UI:**

The flag detail page shows launch overrides inline with conditions:

```
# digest_frequency conditions

| # | Condition                          | Value    | Launch overrides      |
|---|------------------------------------|---------|-----------------------|
| 1 | ctx.plan == PlanLevel.ENTERPRISE   | "daily" |                       |
| 2 | ctx.plan == PlanLevel.PRO          | "weekly" | pro-v2: "daily" (10%) |
| 3 | otherwise                          | "weekly" |                       |
```

The launches management page shows all launches with ramp controls:

```
| Launch   | Dimension | Ramp | Status | Features affected      |
|----------|-----------|------|--------|------------------------|
| pro-v2   | user_id   | 10%  | ACTIVE | notifications, billing |
```

### CLI commands

```
pb launch list [--feature F]     List launches with status and ramp
pb launch ramp <launch> <pct>    Set ramp percentage
pb launch status <launch>        Show launch details and affected flags
pb launch bake <launch>          Set ramp to 100%, status to BAKED
pb launch land <launch>          Promote values and remove launch (see below)
pb launch kill <launch>          Emergency: set status to ABANDONED
pb launch create                 Not needed — launches are defined in YAML config
```

### Mechanical landing: `pb launch land`

The most important CLI command. When a launch is BAKED (100% ramp, baking
period complete), `pb launch land` transforms the config to make the launch
values permanent:

**Algorithm:**

1. Verify the launch status is BAKED.
2. For each config file that references the launch:
   a. For each condition with a `launch: {<launch>: <value>}` override:
      - Replace the condition's `value` with the launch override value.
      - Remove the launch key from the `launch:` map.
      - If the `launch:` map is now empty, remove the `launch:` key entirely.
   b. For each static flag with a `launch: {<launch>: <value>}` override:
      - Replace `value` with the launch override value.
      - Remove the `launch:` key.
3. Delete the launch definition:
   - If feature-scoped: remove the entry from the `launches:` section.
     If the `launches:` section is now empty, remove it.
   - If cross-feature: delete `launches/<launch>.yaml`.
4. Commit the changes and open a PR via `gh`.

**Example transform:**

Before:
```yaml
launches:
  daily-digest-rollout:
    dimension: user_id
    ramp_percentage: 25

flags:
  digest_frequency:
    conditions:
      - when: 'ctx.plan == PlanLevel.PRO'
        value: "weekly"
        launch:
          daily-digest-rollout: "daily"
      - otherwise: "weekly"
```

After `pb launch land daily-digest-rollout`:
```yaml
flags:
  digest_frequency:
    conditions:
      - when: 'ctx.plan == PlanLevel.PRO'
        value: "daily"
      - otherwise: "weekly"
```

The transform is purely mechanical — YAML key replacement and deletion. No
CEL parsing, no semantic analysis, no risk of behavior change. The resulting
config produces identical evaluations to a 100% ramp.

**PR generation:** `pb launch land` creates a branch, commits the
transformed config, and opens a PR. The PR description lists every flag
and condition that changed. The PR is reviewed like any config change —
the diff is small and obvious because only values change.

### Proto changes

Update `EvaluationSource` to clarify the launch source:

```protobuf
EVALUATION_SOURCE_LAUNCH = 9;  // Matched a launch override on a condition
```

The `CompiledLaunch` message in `bundle.proto` is simplified — it no longer
carries `flag_id`, `population_cel`, or `value_json` (those are now in the
condition chain):

```protobuf
message CompiledLaunch {
  string launch_id = 1;
  string dimension = 3;
  int32 ramp_percentage = 6;
}
```

### Validation rules

The sync tool enforces:

1. **Launch exists**: Every `launch:` reference on a condition must resolve
   to a defined launch (in the same feature config or in `launches/`).
2. **Scope enforcement**: Feature-scoped launches cannot be referenced from
   other features.
3. **Dimension is hashable**: The launch dimension must have
   `hashable: true` in proto `DimensionOptions`.
4. **Type compatibility**: Launch override values must match the flag's
   proto type.
5. **No duplicate launch IDs**: A launch ID cannot be defined in both a
   feature config and `launches/`, or in multiple feature configs.
6. **`ramp_percentage` range**: 0–100.
7. **Multi-launch ordering warning**: Conditions with multiple launch
   overrides get a sync warning about ordering sensitivity.

## Phase 2: Launch state in CEL

Phase 1 (above) covers per-condition value overrides — the common case
where a launch changes what a condition returns. Phase 2 adds the ability
to use launch ramp state directly in CEL expressions, enabling structural
changes to the condition chain under a launch.

### Motivation

Some launches need to do more than override a value — they need to add new
conditions or change the structure of evaluation:

```yaml
# "Under the launch, add a new EU-specific price tier that doesn't
#  exist in the base config."
flags:
  monthly_price:
    conditions:
      - when: 'launch.in_ramp("eu-pricing") && ctx.region == Region.EU'
        value: 24.99
      - when: 'ctx.plan == PlanLevel.PRO'
        value: 29.99
      - otherwise: 9.99
```

This can't be expressed with per-condition value overrides because it
introduces a new condition that doesn't exist in the base chain.

### CEL integration

Add a `launch` variable to the CEL environment:

```
launch.in_ramp(launch_id: string) → bool
```

Returns true if the current entity is in the named launch's ramp (i.e.,
`hash(launch_id + dim_value) % 100 < ramp_pct`). The launch must be
ACTIVE or BAKED.

The function is pure and deterministic — same entity always gets the same
result for a given ramp percentage. This preserves the condition chain's
"pure function of context" property with one extension: the evaluation
also depends on operational state (the launch ramp percentage in the DB).

### Caching impact

`launch.in_ramp("X")` in a CEL expression is equivalent to the Phase 1
cache key component: a binary in/out projection. The dimension classifier
recognizes `launch.in_ramp(...)` calls and classifies them as bounded (×2
cardinality per launch reference).

### Mechanical landing with CEL simplification

Landing a Phase 2 launch requires CEL-level transformation, not just YAML
key replacement. At 100% ramp, `launch.in_ramp("X")` is always true.
The landing algorithm:

1. **Substitute**: Replace `launch.in_ramp("<launch>")` with `true` in
   every CEL expression that references the launch.
2. **Fold**: Run the `cel-go` `StaticOptimizer` with
   `NewConstantFoldingOptimizer` to simplify expressions:
   - `true && ctx.region == Region.EU` → `ctx.region == Region.EU`
   - `true || X` → `true`
   - `false && X` → `false`
3. **Prune**: Remove conditions whose expressions folded to `false`
   (they can never match). Merge conditions whose expressions folded
   to `true` with earlier conditions (they always match, shadowing
   everything below).
4. **Unparse**: Convert simplified ASTs back to CEL strings via
   `cel.AstToString()`.
5. **Write back**: Update the YAML config with simplified expressions.
   Remove the launch definition.

**Implementation uses first-party cel-go tooling:**

```go
// 1. Custom ASTOptimizer: replace launch.in_ramp("X") → true
type launchSubstituter struct {
    launchID string
    result   bool // true for landing, false for abandoning
}

func (s *launchSubstituter) Optimize(ctx *cel.OptimizerContext, a *ast.AST) *ast.AST {
    ast.MatchDescendants(a, func(node ast.NavigableExpr) bool {
        if node.Kind() == ast.CallKind &&
           node.AsCall().FunctionName() == "in_ramp" {
            args := node.AsCall().Args()
            if len(args) == 1 && args[0].Kind() == ast.LiteralKind {
                if args[0].AsLiteral().Value().(string) == s.launchID {
                    ctx.UpdateExpr(node, ctx.NewLiteral(types.Bool(s.result)))
                }
            }
        }
        return false
    })
    return a
}

// 2. Chain with constant folder
folder, _ := cel.NewConstantFoldingOptimizer()
opt, _ := cel.NewStaticOptimizer(&launchSubstituter{launchID: "eu-pricing", result: true}, folder)
optimized, _ := opt.Optimize(env, checkedAST)
simplified := cel.AstToString(optimized)
```

**Example transform:**

Before:
```yaml
conditions:
  - when: 'launch.in_ramp("eu-pricing") && ctx.region == Region.EU'
    value: 24.99
  - when: 'ctx.plan == PlanLevel.PRO'
    value: 29.99
  - otherwise: 9.99
```

After `pb launch land eu-pricing`:
```yaml
conditions:
  - when: 'ctx.region == Region.EU'
    value: 24.99
  - when: 'ctx.plan == PlanLevel.PRO'
    value: 29.99
  - otherwise: 9.99
```

The `launch.in_ramp("eu-pricing")` was replaced with `true`, the folder
simplified `true && ctx.region == Region.EU` to `ctx.region == Region.EU`,
and the result was unparsed back to a clean expression.

### Phase 2 validation rules

The sync tool adds:

1. **`launch.in_ramp()` references resolve**: Every `launch.in_ramp("X")`
   call must reference a defined launch.
2. **Population dimension restriction**: CEL expressions containing
   `launch.in_ramp()` may reference unbounded dimensions freely (the
   launch hash itself is the bounded projection). However, `launch.in_ramp()`
   should appear in conjunction (`&&`) with other predicates, not as a
   standalone condition — a standalone `launch.in_ramp("X")` is just a
   Phase 1 override and should use the simpler syntax.

### Interaction between Phase 1 and Phase 2

A single launch can use both mechanisms:

```yaml
flags:
  monthly_price:
    conditions:
      # Phase 2: new condition that only exists under the launch
      - when: 'launch.in_ramp("pro-v2") && ctx.region == Region.EU'
        value: 24.99
      # Phase 1: existing condition with value override
      - when: 'ctx.plan == PlanLevel.PRO'
        value: 29.99
        launch:
          pro-v2: 19.99
      - otherwise: 9.99
```

The landing transform handles both:
- Phase 1 overrides: YAML key replacement (as described above)
- Phase 2 expressions: CEL substitution + folding

Both transforms run in the same `pb launch land` invocation.

## Migration from current implementation

The current launch implementation (migration 007, `launch.go`,
`configfile.go`, `sync/conditions.go`) stores a flat value per launch
and evaluates launches as a separate step before conditions. Migration:

1. **Schema migration**: Drop `flag_id`, `population_cel`, `value`
   columns from `launches` table. Add `description` column.
2. **Config format**: The YAML `launches:` section changes from
   `{flag, dimension, population, value, ramp_percentage}` to
   `{dimension, ramp_percentage, description}`. Launch-to-flag binding
   moves inline to conditions.
3. **Sync pipeline**: `SyncConditions` encodes launch overrides into
   the `StoredCondition.Launches` map instead of the `launches` table's
   `value` column.
4. **Evaluator**: Launch evaluation moves from a separate
   `EvaluateLaunches()` step to inline within condition evaluation.
   The `CachedLaunch` struct drops `Population` and `Value` fields.
5. **Admin store**: Query changes to reflect the simplified launch table.
6. **Admin UI**: Conditions table gains a "Launch overrides" column.
   Separate launches table simplified.

No data migration is needed for existing launches — they should be
completed or abandoned before upgrading since the value semantics change.

## Future work

### Approval gates

Launches gain an optional approval workflow:

```yaml
# launches/pro-v2.yaml
dimension: user_id
ramp_percentage: 0
approval:
  required: true
  approvers:
    - product-team
    - eng-leads
```

`pb launch ramp` requires approval before the ramp can be increased.
The approval state is tracked in the database and surfaced in the admin UI.

### Staged ramp plans

A launch can define a ramp schedule:

```yaml
ramp_plan:
  - percentage: 1
    bake: 24h
  - percentage: 10
    bake: 48h
  - percentage: 50
    bake: 72h
  - percentage: 100
```

The CLI or a scheduled job advances the ramp according to the plan,
pausing at each stage for the bake duration. Automatic advancement can
be gated on health metrics (error rate, latency) if integrated with
monitoring.

### Experiments

Experiments build on the launch primitive but assign entities to named
variants instead of binary in/out ramp:

```yaml
experiments:
  notification-cadence:
    dimension: user_id
    variants:
      control:
        percentage: 50
      daily:
        percentage: 25
      realtime:
        percentage: 25
```

Variant assignment uses the same hash mechanism (`hash(experiment_id +
dim_value) % 100`) but maps to variant buckets instead of a single ramp
threshold. The CEL integration (Phase 2) naturally extends:
`experiment.variant("notification-cadence")` returns the variant name.

Experiment logging is handled by the generated client SDK via an optional
`ExperimentLogger` callback — the evaluator remains stateless.
